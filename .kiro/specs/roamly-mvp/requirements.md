# Requirements Document

## Introduction

Roamly is an AI-powered travel planning SaaS application with the tagline "Plan Less. Explore More." The application enables users to describe their travel preferences through a guided questionnaire and receive a fully personalized vacation itinerary generated by an AI model. The MVP covers user authentication, profile management, preference collection, AI itinerary generation, and trip persistence — with no booking or payment features. The architecture must be extensible to support multiple AI providers in the future.

This document covers the architecture, development roadmap, and MVP feature requirements. It does not include implementation code — that begins after approval.

---

## Glossary

- **System**: The Roamly web application as a whole
- **User**: An authenticated person interacting with the System
- **Auth_Service**: The Supabase Auth subsystem responsible for identity management
- **Profile_Service**: The service responsible for reading and writing User profile data
- **Trip_Service**: The service responsible for creating, reading, and persisting Trip records
- **AI_Service**: The provider-agnostic AI integration layer responsible for generating itineraries
- **AI_Provider**: An external large-language-model API (e.g., Google Gemini) invoked by the AI_Service
- **Questionnaire**: The multi-step preference collection flow presented to the User before itinerary generation
- **Itinerary**: A structured day-by-day travel plan generated by the AI_Service based on User preferences
- **Trip**: A persisted record combining a completed Questionnaire response and its associated Itinerary
- **Preference_Set**: The full set of answers collected by the Questionnaire, including budget, destination, travel style, food preferences, accommodation type, transportation preference, activity preferences, group size, and trip duration
- **Session**: An authenticated browser session maintained by the Auth_Service
- **ORM**: Prisma, the object-relational mapper used to interact with the Supabase PostgreSQL database
- **Folder_Architecture**: The scalable project directory structure separating components, pages/app, services, lib, hooks, types, utils, constants, database, api, and ai subdirectories

## Requirements

### Requirement 1: Project Architecture and Folder Structure

**User Story:** As a developer, I want a well-defined, scalable folder architecture, so that the codebase remains maintainable as the feature set grows.

#### Acceptance Criteria

1. THE System SHALL organize source code under a root `src/` directory containing at minimum the following top-level subdirectories: `app/`, `components/`, `services/`, `lib/`, `hooks/`, `types/`, `utils/`, `constants/`, `db/`, `api/`, and `ai/`; additional subdirectories are permitted.
2. THE System SHALL place all Next.js App Router page and layout files exclusively under `src/app/`.
3. THE System SHALL place all reusable React components under `src/components/`, further organized into `ui/`, `layout/`, and `features/` subdirectories.
4. THE System SHALL place all server-side business logic and external integrations under `src/services/`, with each file serving exactly one domain entity (e.g., `tripService.ts` serves only Trip-related logic and does not contain Profile-related logic).
5. THE System SHALL place all AI provider integrations under `src/ai/`.
6. THE System SHALL place all Prisma schema files and database seed scripts under `src/db/`.
7. THE System SHALL place all shared TypeScript type and interface definitions under `src/types/`.
8. THE System SHALL place all application-wide constants (route names, API endpoint paths, default values) under `src/constants/`.

### Requirement 2: Technology Setup and Coding Standards

**User Story:** As a developer, I want a consistent technology setup and coding standards document, so that all contributors produce uniform, high-quality code.

#### Acceptance Criteria

1. THE System SHALL be built with Next.js (App Router), React, TypeScript (strict mode), Tailwind CSS, and shadcn/ui as the primary UI component library.
2. THE System SHALL use Supabase PostgreSQL as the sole persistent data store, accessed via Prisma ORM.
3. THE System SHALL use Supabase Auth for all authentication flows, with no custom credential storage.
4. THE System SHALL deploy on Vercel, with environment variables managed through Vercel's environment configuration.
5. THE System SHALL enforce TypeScript strict mode across all source files; any use of the `any` type SHALL be accompanied by both an `// eslint-disable-next-line @typescript-eslint/no-explicit-any` comment and an inline comment of at least 10 characters explaining why `any` is necessary in that specific location.
6. THE System SHALL follow the naming conventions: PascalCase for React components and TypeScript interfaces; camelCase for variables, functions, and file names (except components); kebab-case for CSS class names and route segments.
7. THE System SHALL use named exports for all utility functions and hooks, and default exports only for Next.js page and layout components.
8. WHEN any source file introduces an HTTP or WebSocket call to a third-party URL not already covered by an existing service module under `src/services/`, THE System SHALL define a new service module with a TypeScript interface for that integration before any implementation code making that call is merged.

### Requirement 3: Database Schema Planning

**User Story:** As a developer, I want a clearly planned database schema, so that all MVP data models are defined before implementation begins.

#### Acceptance Criteria

1. THE System SHALL define the following Prisma models in `src/db/schema.prisma`: `User`, `Profile`, `Trip`, and `PreferenceSet`.
2. THE `User` model SHALL store: `id` (UUID, primary key), `email` (String, unique, required), `createdAt` (DateTime, required), and `updatedAt` (DateTime, required).
3. THE `Profile` model SHALL store: `id` (UUID, primary key), `userId` (UUID, foreign key to `User`, required), `displayName` (String, required, max 100 characters), `avatarUrl` (String, optional/nullable), `createdAt` (DateTime, required), and `updatedAt` (DateTime, required).
4. THE `Trip` model SHALL store: `id` (UUID, primary key), `userId` (UUID, foreign key to `User`, required), `title` (String, required, max 200 characters), `status` (enum `TripStatus` with values `DRAFT` and `COMPLETE`, required, default `DRAFT`), `itineraryJson` (JSON, optional/nullable), `createdAt` (DateTime, required), and `updatedAt` (DateTime, required).
5. THE `PreferenceSet` model SHALL store: `id` (UUID, primary key), `tripId` (UUID, foreign key to `Trip`, required), `destination` (String, optional/nullable, max 200 characters), `budget` (Int, optional/nullable, range 0–999,999,999 USD), `travelStyle` (String, optional/nullable), `foodPreferences` (String[], optional, default empty array), `accommodationType` (String, optional/nullable), `transportationPreference` (String, optional/nullable), `activityPreferences` (String[], optional, default empty array), `groupSize` (Int, optional/nullable, range 1–500), `durationDays` (Int, optional/nullable, range 1–365), `createdAt` (DateTime, required), and `updatedAt` (DateTime, required).
6. THE System SHALL enforce a one-to-one relationship between `Profile` and `User`; deleting a `User` record SHALL cascade-delete the associated `Profile` record. THE System SHALL enforce a one-to-many relationship between `User` and `Trip`; deleting a `User` record SHALL cascade-delete all associated `Trip` records.
7. THE System SHALL enforce a one-to-one relationship between `Trip` and `PreferenceSet`; deleting a `Trip` record SHALL cascade-delete the associated `PreferenceSet` record.

### Requirement 4: User Authentication — Email

**User Story:** As a visitor, I want to register and log in with my email and password, so that I can access my personal travel plans securely.

#### Acceptance Criteria

1. WHEN a visitor submits a registration form with an email address that matches the RFC 5322 format and a password that meets all password rules, THE Auth_Service SHALL create a new User account and send an email verification link valid for 24 hours.
2. WHEN a visitor submits an email that is already registered, THE Auth_Service SHALL return an error message indicating the email is already in use.
3. WHEN a visitor submits a verified email and correct password on the login form, THE Auth_Service SHALL create an authenticated Session and redirect the User to the dashboard.
4. IF a visitor submits an unverified email on the login form, THEN THE Auth_Service SHALL return an error message instructing the visitor to verify their email before logging in, without granting a Session.
5. IF a visitor submits an incorrect password, THEN THE Auth_Service SHALL return a generic error message that does not confirm or deny whether the email address is registered in the system.
6. WHEN an authenticated User requests logout, THE Auth_Service SHALL invalidate the Session and redirect the User to the landing page without displaying any error or status messages.
7. THE Auth_Service SHALL enforce a minimum password length of 8 characters and a maximum password length of 128 characters during registration; passwords outside this range SHALL be rejected with a validation error before any network request is made.
8. IF a visitor clicks an expired email verification link, THEN THE Auth_Service SHALL display a message explaining the link has expired and offer the visitor a way to request a new verification email.

### Requirement 5: User Authentication — Google OAuth

**User Story:** As a visitor, I want to log in with my Google account, so that I can access Roamly without creating a separate password.

#### Acceptance Criteria

1. WHEN a visitor clicks "Continue with Google," THE Auth_Service SHALL initiate an OAuth 2.0 authorization flow with Google using the configured OAuth client credentials.
2. WHEN Google returns a successful authorization code, THE Auth_Service SHALL exchange the code for tokens, then create a new User record if one does not already exist for the returned Google account identifier, establish an authenticated Session, and redirect the User to the dashboard.
3. IF the Google authorization flow fails due to a Google-side error or network failure, THEN THE Auth_Service SHALL display an error message describing the failure type and return the visitor to the login page without creating a Session.
4. IF the visitor cancels the Google authorization flow (e.g., closes the popup or clicks "Cancel" on the Google consent screen), THEN THE Auth_Service SHALL return the visitor to the login page without creating a Session and without displaying an error message.
5. WHEN a Google-authenticated User completes their first successful login (i.e., a new User record was just created), THE Profile_Service SHALL create a Profile record with `displayName` set to the value returned by Google and `avatarUrl` set to the profile picture URL returned by Google; IF Google does not return a display name, `displayName` SHALL be set to the User's email address prefix (the portion before `@`).

### Requirement 6: User Profile Management

**User Story:** As an authenticated User, I want to view and edit my profile, so that my name and avatar are correct across the application.

#### Acceptance Criteria

1. WHEN an authenticated User navigates to the profile page, THE Profile_Service SHALL retrieve and display the User's current `displayName` and `avatarUrl`; IF `avatarUrl` is null or unset, THE System SHALL display a default placeholder avatar image; IF `displayName` is null or unset, THE System SHALL display an empty string in the display name field.
2. WHEN an authenticated User submits an updated display name that contains between 1 and 64 non-whitespace characters (after trimming leading and trailing whitespace), THE Profile_Service SHALL persist the trimmed value and display a success confirmation message.
3. IF an authenticated User submits a display name that is empty, contains only whitespace, or exceeds 64 characters after trimming, THEN THE Profile_Service SHALL return a validation error identifying the violated constraint without persisting the change.
4. WHEN an authenticated User uploads a new avatar image of type JPEG, PNG, or WebP with a file size of 5 MB or less, THE Profile_Service SHALL store the file, update `avatarUrl` to the new file's URL, and display a success confirmation only after both the file storage and the database update complete successfully.
5. IF an authenticated User uploads a file that exceeds 5 MB or is not of type JPEG, PNG, or WebP, THEN THE Profile_Service SHALL return a validation error identifying the specific violated constraint (size or type), reject the upload without storing any data, and not display a success message.
6. IF the file storage operation fails during an avatar upload (e.g., storage service error), THEN THE Profile_Service SHALL return an error message indicating the upload failed, leave `avatarUrl` unchanged, and not display a success message.
7. IF the database update fails after a successful file storage during avatar upload, THEN THE Profile_Service SHALL attempt to delete the newly stored file and return an error message indicating the update failed, without displaying a success message.
8. IF the database write fails when persisting an updated display name, THEN THE Profile_Service SHALL return an error message indicating the update failed without displaying a success message.

### Requirement 7: Trip Creation

**User Story:** As an authenticated User, I want to start a new trip, so that I can begin planning a vacation.

#### Acceptance Criteria

1. WHEN an authenticated User initiates trip creation, THE Trip_Service SHALL create a new `Trip` record with `status` set to `DRAFT` and associate it with the User's `id`.
2. WHEN a `DRAFT` Trip is successfully created, THE System SHALL redirect the User to the Questionnaire for that Trip in the same navigation response.
3. THE Trip_Service SHALL allow an authenticated User to have up to 10 concurrent `DRAFT` Trips.
4. IF an authenticated User attempts to create a new Trip while already having 10 `DRAFT` Trips, THEN THE Trip_Service SHALL reject the request and display an error message instructing the User to complete or delete an existing draft before creating a new one.
5. WHEN an authenticated User requests deletion of a Trip they own, THE Trip_Service SHALL remove the `Trip` record and return a success response to the caller.
6. WHEN the `Trip` record is successfully removed, THE Trip_Service SHALL attempt to remove the associated `PreferenceSet` record; IF the `PreferenceSet` removal fails, THE Trip_Service SHALL log the failure but SHALL NOT surface an error to the User or block the success response already returned.
7. IF an unauthenticated or unauthorized request is made to delete a Trip, THEN THE Trip_Service SHALL reject the request with a 403 response and take no action on any records.

### Requirement 8: Preference Questionnaire

**User Story:** As an authenticated User, I want to answer a guided multi-step questionnaire about my travel preferences, so that the AI can generate a relevant itinerary.

#### Acceptance Criteria

1. THE System SHALL present the Questionnaire as a multi-step flow with one topic per step, covering in order: Destination, Budget, Trip Duration, Group Size, Travel Style, Accommodation Type, Transportation Preference, Food Preferences, and Activity Preferences — totaling 9 steps.
2. WHEN a User advances to the next Questionnaire step, THE System SHALL validate that the current step's required field is populated and meets its constraints; IF validation fails, THE System SHALL display a descriptive error message on the current step and SHALL NOT advance to the next step.
3. WHEN a User navigates back to a previous Questionnaire step, THE System SHALL restore all previously entered answers exactly as the User left them.
4. THE System SHALL present selection-based steps using interactive cards or chips with the following cardinality: Travel Style — single-select; Transportation Preference — single-select; Accommodation Type — multi-select; Food Preferences — multi-select; Activity Preferences — multi-select.
5. THE System SHALL present the Budget step using a range slider with a visible numeric label showing the current value, with a selectable range of 100 to 50,000 USD in increments of 100.
6. THE System SHALL present the Group Size step using a numeric stepper control with a minimum value of 1 and a maximum value of 20; the stepper SHALL prevent input outside this range.
7. THE System SHALL present the Trip Duration step using a numeric stepper control with a minimum value of 1 day and a maximum value of 30 days; the stepper SHALL prevent input outside this range.
8. THE System SHALL present the Destination step as a free-text input field; WHEN the User submits the Destination step, THE System SHALL validate that the input contains between 2 and 100 characters after trimming whitespace; IF the input is outside this range, THE System SHALL display a validation error and SHALL NOT advance to the next step.
9. WHEN a User completes all 9 Questionnaire steps and clicks the final submit action, THE System SHALL display a visual confirmation indicator (e.g., a spinner or progress message) and then persist the Preference_Set; only after a successful persistence confirmation SHALL THE System proceed to invoke AI itinerary generation.
10. IF the Preference_Set persistence fails during Questionnaire submission, THEN THE System SHALL display an error message, retain all answers the User entered, and allow the User to retry the submission without re-entering any data.
11. IF a User navigates away from the Questionnaire view or closes the browser tab while the Questionnaire is open and not yet submitted, THEN THE Trip_Service SHALL retain the Trip record in `DRAFT` status; any steps that were answered before navigation SHALL be preserved in the session or local state if the User returns to the same Questionnaire within the same browser session.

### Requirement 9: AI Itinerary Generator

**User Story:** As an authenticated User, I want an AI-generated itinerary based on my preferences, so that I receive a personalized travel plan without manual research.

#### Acceptance Criteria

1. WHEN a Preference_Set containing non-null values for `destination`, `budget`, `durationDays`, and `groupSize` is successfully persisted for a Trip in `DRAFT` status, THE AI_Service SHALL invoke the configured AI_Provider to generate an Itinerary.
2. THE AI_Service SHALL accept a provider configuration that maps a named AI_Provider identifier (e.g., `"gemini"`) to its provider class, such that changing the active provider requires updating only the configuration value — no changes to the AI_Service calling code or the Trip_Service are required.
3. WHEN the AI_Provider returns a non-empty response, THE AI_Service SHALL parse the response into a structured Itinerary object conforming to the following schema: `title` (string), `summary` (string), `days` (array of `DayPlan`), where each `DayPlan` contains `dayNumber` (integer), `theme` (string), and `activities` (array of `Activity`).
4. WHEN the AI_Provider returns a valid, non-empty response that is successfully parsed into a conforming Itinerary object, THE Trip_Service SHALL persist the Itinerary as `itineraryJson` on the Trip record and update the Trip `status` to `COMPLETE`.
5. WHILE an Itinerary is being generated, THE System SHALL display a loading indicator to the User; the loading indicator SHALL remain visible until either the generation succeeds or an error/timeout occurs.
6. IF the AI_Provider returns an error response or does not return a response within 30 seconds, THEN THE AI_Service SHALL hide the loading indicator, display an error message that includes the reason for failure (e.g., "The AI service is unavailable" or "Generation timed out"), and retain the Trip in `DRAFT` status so the User can retry.
7. WHEN generation completes successfully and the Trip status is updated to `COMPLETE`, THE System SHALL hide the loading indicator and navigate the User to the Itinerary detail view for that Trip.
8. IF the AI_Provider response is empty, malformed, or does not conform to the Itinerary schema, THEN THE AI_Service SHALL treat the response as a generation failure: hide the loading indicator, display a descriptive error message, and retain the Trip in `DRAFT` status without updating `itineraryJson` or the Trip `status`.

### Requirement 10: Save and Retrieve Trips

**User Story:** As an authenticated User, I want to revisit my previously generated itineraries, so that I can reference my travel plans at any time.

#### Acceptance Criteria

1. WHEN an authenticated User navigates to the dashboard, THE Trip_Service SHALL retrieve all Trips associated with that User, ordered by `createdAt` descending, and display each as a Trip card showing: `title`, destination (from the associated PreferenceSet, displayed as "—" if null), trip duration formatted as "X days" (from `durationDays` in the PreferenceSet, displayed as "—" if null), and `createdAt` date.
2. WHEN an authenticated User selects a `COMPLETE` Trip from the dashboard, THE Trip_Service SHALL retrieve the associated Itinerary from `itineraryJson` and display it in the Itinerary detail view.
3. WHEN an authenticated User selects a `DRAFT` Trip from the dashboard, THE System SHALL navigate to the Questionnaire for that Trip without displaying an Itinerary.
4. IF an authenticated User has no saved Trips, THEN THE System SHALL display only an empty state message with a prompt to create a new Trip; no Trip card, list header, or pagination control SHALL be visible.
5. WHEN an authenticated User confirms deletion of a Trip from the dashboard, THE Trip_Service SHALL send a delete request; upon receiving a successful response, THE System SHALL remove the corresponding Trip card from the displayed list within 1 second, without a full page reload.
6. IF the Trip retrieval request fails (e.g., network error or server error), THEN THE System SHALL display an error message and a retry action; no partial Trip list SHALL be shown.
7. IF the Trip deletion request fails, THEN THE System SHALL display an error message and leave the Trip card in the list unchanged; no success indication SHALL be shown.

### Requirement 11: Initial UI Design Principles and Wireframe Direction

**User Story:** As a product stakeholder, I want defined UI design principles and wireframe direction, so that all developers build toward the same visual and interaction standard.

#### Acceptance Criteria

1. THE System SHALL implement a mobile-first responsive layout using Tailwind CSS breakpoints such that all interactive controls are reachable and usable without horizontal scrolling on viewport widths from 375px to 1440px and above; specifically, no interactive element SHALL be clipped or require horizontal scroll to activate on viewports within this range.
2. THE System SHALL apply the following concrete UI attributes: card components with rounded corners (border-radius ≥ 8px) and subtle drop shadows; body text set in a sans-serif typeface at a minimum of 16px on mobile; section headings set at a minimum of 24px with font-weight ≥ 600; a minimum of 16px padding inside all interactive card components.
3. THE System SHALL apply CSS transitions with a duration between 150ms and 300ms on all interactive elements (buttons, cards, chips) that change visual state on user interaction (hover, focus, active).
4. THE System SHALL maintain a minimum color contrast ratio of 4.5:1 between text color and background color for all text elements rendered in the UI, including body copy, labels, placeholders, and error messages.
5. WHEN the application is initialized, THE System SHALL have a defined Tailwind CSS configuration that includes at minimum: a `colors` key with at least one primary color token, a `fontFamily` key, a `fontSize` scale, a `spacing` scale, and a `boxShadow` key with at least one named shadow token; no component SHALL be built before this configuration is committed.
6. WHERE content requires a network fetch before it can be displayed and the fetch duration is 200ms or longer (Trip list, Itinerary view), THE System SHALL display animated skeleton placeholder elements that match the layout dimensions of the expected content; THE System SHALL NOT display skeleton placeholders during error states or when content is available synchronously.

### Requirement 12: Development Roadmap

**User Story:** As a project manager, I want a phased development roadmap for the MVP, so that work can be sequenced and tracked clearly.

#### Acceptance Criteria

1. THE System SHALL be developed in the following sequential phases, where each phase's deliverables must be present and verifiable (i.e., the relevant files, routes, or database migrations exist and all automated checks pass) before the next phase begins:
   - **Phase 0 – Foundation**: Repository setup, folder architecture, Prisma schema, Tailwind/shadcn/ui configuration, environment variable setup, Vercel project initialization.
   - **Phase 1 – Authentication**: Supabase Auth integration, email/password registration and login, Google OAuth, session management, protected route middleware.
   - **Phase 2 – Profile**: Profile page, display name editing, avatar upload.
   - **Phase 3 – Trip & Questionnaire**: Trip creation flow, multi-step Questionnaire UI, Preference_Set persistence.
   - **Phase 4 – AI Itinerary Generation**: AI_Service abstraction layer, Gemini provider, prompt engineering, Itinerary parsing, loading states, error handling.
   - **Phase 5 – Dashboard & Trip History**: Trip list, Trip detail view, delete Trip, empty state.
   - **Phase 6 – Polish & QA**: Animation refinement, accessibility audit, performance optimization, responsive testing, end-to-end tests.
2. THE System SHALL not begin Phase 1 until all Phase 0 deliverables are merged to the main branch and all automated checks (linting, type-checking, and any configured CI steps) pass without errors.
3. WHEN a phase is complete, THE System SHALL have 100% of the automated tests for that phase passing (zero failing, zero erroring tests) before the next phase begins.
4. A phase SHALL be considered complete only when all deliverables listed for that phase are present in the repository, all relevant automated checks pass, and any database migrations for that phase have been applied to the development environment.

### Requirement 13: AI Provider Abstraction

**User Story:** As a developer, I want the AI integration layer to be provider-agnostic, so that future AI providers can be swapped in without rewriting calling code.

#### Acceptance Criteria

1. THE AI_Service SHALL define a `GenerateItineraryRequest` TypeScript type containing all Preference_Set fields required for prompt construction: `destination`, `budget`, `durationDays`, `groupSize`, `travelStyle`, `accommodationType`, `transportationPreference`, `foodPreferences`, and `activityPreferences`.
2. THE AI_Service SHALL define a `GenerateItineraryResponse` TypeScript type containing the structured Itinerary object with fields: `title` (string), `summary` (string), and `days` (array of `DayPlan`).
3. THE AI_Service SHALL define an `AIProvider` TypeScript interface with a single method: `generateItinerary(request: GenerateItineraryRequest): Promise<GenerateItineraryResponse>`.
4. THE System SHALL include a concrete `GeminiProvider` class that implements the `AIProvider` interface using the Google Gemini API.
5. THE AI_Service SHALL select the active `AIProvider` implementation based on the value of the `AI_PROVIDER` environment variable (server-side only); IF the variable is not set or is set to an unrecognized value, THE AI_Service SHALL default to the `GeminiProvider`.
6. WHEN a different `AIProvider` implementation is substituted in configuration, THE Trip_Service SHALL require no code changes to continue invoking itinerary generation; any new `AIProvider` implementation SHALL accept a `GenerateItineraryRequest` and return a `GenerateItineraryResponse` conforming to the same TypeScript types defined in criteria 1 and 2.
7. THE `GeminiProvider` SHALL construct a prompt that includes all non-null fields from the `GenerateItineraryRequest` and explicitly instructs the model to return a response formatted as a JSON object conforming to the `GenerateItineraryResponse` schema.
8. THE `GeminiProvider` SHALL validate the AI_Provider response against the `GenerateItineraryResponse` JSON schema before returning; IF the response does not conform to the schema, THE `GeminiProvider` SHALL throw an error with a message that identifies which schema fields are missing or invalid.
9. IF the `GeminiProvider` receives a response that fails schema validation, THEN THE AI_Service SHALL propagate the error to the caller as a rejected Promise; THE AI_Service SHALL NOT silently swallow the validation error or return a partial Itinerary object.
